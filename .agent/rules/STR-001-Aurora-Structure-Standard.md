# Team Protocol: STR-001 - Repository Structure Standard

## 1. 3層構造 (Ag-Mng-Prd) の厳守
プロジェクトは以下の 3 層で物理的かつ論理的に分離されなければならない。
- **.agent/** (システム層): AI の知能、スキル、テンプレート、ルール。
- **management/** (ビジネス層): 要件、計画、実装履歴、ポータル。
- **product/** (製品層): ソースコード、アセット、package.json等。直下のファイル散在を禁じ、原則このフォルダ内に完結させる。
- **(System/Implicit)**: .agent, node_modules。これら以外のフォルダ作成は原則禁止とする。

## 2. パス解決の原則
- **Root-Relative**: すべてのスクリプト・スキルは、プロジェクトルートを基点とした動的なパス解決を行わなければならない。
- **Anti-Hardcode**: 環境に依存する絶対パスのハードコードを禁止する。

## 3. 情報の階層化・グループ化設計
情報の「アクセシビリティ（到達しやすさ）」と「体系性」を最大化する。
- **Logical Grouping**: ポータル表示や管理ドキュメントにおいて、時系列（Phase 01, 02...）のレポートは個別に散布させず、必ず `Implementation` 等の共通セクション配下へ論理的に集約すること。
- **Fixed Anchor**: マスターレポート（00_Master）は、プロジェクトの最高意志決定記録として、常にポータルの初期表示および最上位に固定されなければならない。

## 4. エントロピー（無秩序）の抑制
- **Folder Cap**: プロジェクトルートには、上記主要3層以外のフォルダを生成してはならない。
- **Single Source of Truth**: レポートや記録は必ず `management` 配下の指定位置へ移動させ、ルートに作業残渣を残さないこと。
