# 📜 Documentation Standard (チーム反映用)

## 1. 画像とアセットのパス
- **原則**: すべて **相対パス (`./` または `../`)** で記述すること。
- **禁止事項**: `C:/Users/...` や `file:///...` などの絶対パスは、通常のMarkdownファイルでは使用を禁止する。
- **Antigravityアーティファクト特例**:
    - **Artifact Viewer**（Issue LogやReport等）で確実に画像を表示させる際は、規約に基づき **先頭スラッシュ付き絶対パス (`/C:/Users/...)** を使用すること。
    - また、参照画像は事前に **アーティファクトディレクトリ内へコピー** した上で、そのパスを指定すること。

## 2. 図面（ダイアグラム）の管理
- **原則**: ワークスペース内の `.md` ファイルにおいては、Mermaid などの動的コードブロックによる表示を避け、**静的画像 (PNG/SVG)** を埋め込むこと。
- **理由**: レンダリングエンジン（Antigravity Artifactビューア等）の互換性問題を確実に回避し、全ユーザーに同一の進捗情報を届けるため。

## 3. フェーズ管理の 3-Tier ドキュメント
- **原則**: 各開発フェーズ（Phase XX）は必ず以下の3つのドキュメントで構成すること。
    1. **Plan.md**: 実装開始前の詳細設計、タスク分割、および **該当範囲をハイライトしたガントチャート**。
    2. **Report.md**: 遂行プロセス、**対象タスクの完了ステータス（進捗率）**、および **視覚的エビデンス（成果物画像）** の記録。
    3. **Verification_Log.md**: 品質検証結果と最終合格証跡。

## 5. ドキュメントポータルの義務化
- **原則**: プロジェクト完了時には必ず `🌟_Portal_Generator_Master` を実行し、`08_Documentation_Portal/index.html` を最新状態にすること。
- **目的**: 階層化された大量のドキュメントを、専門知識のない閲覧者でもブラウザ上でシームレスに確認可能にする。

## 6. Zero-Config & Fallback UI
- **原則**: アプリケーション本体（ルートの index.html）がサーバー無しで動作しない場合、必ず `file://` プロトコルで開かれた際の「案内画面（Fallback UI）」を実装すること。
- **実装内容**: ポータルサイトへのリンク、および `npm run dev` 等の起動コマンドの明示。
