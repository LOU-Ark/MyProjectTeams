# Team Protocol: QLT-001 - Quality Verification Protocol

**カテゴリ**: 品質保証 (QA)
**適用範囲**: 全コード修正、新機能実装、バグ修復

## 1. 三段階検証の義務付け
いかなる小さな修正であっても、以下のステップを順に踏み、その証跡を確保しなければならない。

### ① 静的検証 (Static Analysis)
- **目的**: 構文、型、パスの整合性を「実行前」に担保する。
- **必須アクション**:
    - 出力物のソースコード点検（エスケープ漏れ、文字化けの確認）。
    - 依存関係の物理的存否確認。
    - Linter または Syntax Check による文法確認。

### ② 統合検証 (Integration Testing)
- **目的**: モジュール間の干渉や、データの流し込みによる破壊を防ぐ。
- **必須アクション**:
    - 境界値（空データ、巨大データ、特殊文字）を用いたモジュール単体テスト。
    - テンプレートインターポレーション（${} 等）の干渉チェック。

### ③ E2E検証 (System/Evidence Testing)
- **目的**: 最終的なユーザー体験（UX）が意図通りであることを証明する。
- **Evidence Linkage**: すべての実装フェーズ報告書には、`04_Quality_Gate` に保存されたエビデンスへのリンクが必須である。
- **必須アクション**:
    - 最終成果物のパターンマッチング（期待される文字列の存在確認）。
    - 視覚的エビデンス（ブラウザエージェントによる描画確認）の取得。

## 2. 完了報告の定義
「修復しました」という報告は、上記 3 ステップの証跡（ターミナル出力、ログ、画像）を伴うもののみが受理され、主観的な「動くはず」という報告は**却下（Reject）**の対象となる。

## 3. インシデント・ハンドリング (3-Fail-Pivot)
自動検証（Playwright等）において、3回以上の試行が失敗し、かつ原因が環境依存であると判断される場合、以下の行動を取らなければならない。
1.  **Stop & Analysis**: 即座に作業を停止し、発生状況を `05_Issue_Log` に記録する。
2.  **Pivot to Alternative**: ブラウザ操作が不可であっても、`curl` や `node` スクリプト等による**サーバー/APIレベルの生存確認・機能検証**を代替として実施し、最低限の機能整合性を証明すること。
3.  **Governance Report**: 代替検証の結果と環境エラーの因果関係を詳細に考察し、ユーザーに判断を仰ぐこと。

## 4. 納品条件の厳格化
いかなる理由があっても、対応するフェーズの「合格判定（Pass）」を含む品質レポートなしに、納品（05_Delivery）フェーズを開始してはならない。

---
**遵守義務**: 本プロトコルを無視した「直感によるパッチ当て」や「検証の省略」は、チームの信頼を損なう重大な規約違反と見なす。
