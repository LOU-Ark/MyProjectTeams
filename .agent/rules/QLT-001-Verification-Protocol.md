# RULE: QLT-001 - Quality Verification Protocol (3-Tier)

> [🏠 00_Master ] > [ .agent ] > [ rules ] > QLT-001-Verification-Protocol

## 1. 三段階検証の義務付け
いかなる修正であっても、以下のステップを順に踏み、その証跡を `04_Quality_Gate` に確保しなければならない。証跡なき完了報告は客観的な「規約違反」とみなす。

### ① 静的検証 (Static Analysis)
- **目的**: 構文、型、パスの整合性を「実行前」に担保。
- **Action**: 構文チェック、文字化け確認、物理的なパス存在確認。

### ② 統合検証 (Integration Testing)
- **目的**: データの受け渡しや、既存機能への破壊的影響の確認。
- **Action**: 境界値テスト、コンポーネント間の連携確認。

### ③ E2E検証 (System/Evidence Testing)
- **目的**: 最終的なユーザー体験（UX）の保証。
- **Action**: ブラウザエージェントによる描画確認、期待される文字列のパターンマッチング。

## 2. 完了報告の定義 (No Fake Pass)
「修復したはず」「動くはず」という主観的な報告は却下される。
- **Evidence Linkage**: フェーズレポートには、検証ログ（.md）または証跡画像へのリンクが必須である。
- **Block Completion**: 検証に失敗している、あるいはエビデンスが不足している場合、ダッシュボードの 100% 更新および「フェーズ完了」の宣言を行ってはならない。

## 3. インシデント・ハンドリング (3-Fail-Pivot)
自動検証において 3 回連続で失敗し、かつ環境依存が明確な場合は、`05_Issue_Log` に記録し、代替検証（curl 等）にピボットする権限を有する。
