# 🌟 08 Governance & Structure Optimization Workflow

## 📌 目的
ユーザーからのフィードバックに基づき、プロジェクトの「構造」「規約」「構築ロジック（.agent）」を自律的に進化させ、その最適化された状態を恒久的に固定（ハードニング）することを目的とする。

## 🛠️ フェーズ定義

### Phase 1: Insight Capture (指摘の受領)
- [ ] ユーザーの指摘内容を、単なる「個別修正」ではなく「システムの課題」として捉える。
- [ ] 影響範囲を特定する（ディレクトリ構造、規約、ビルドスクリプト、アセット等）。

### Phase 2: Systemic Action (自律的改修)
- [ ] **Step 1: 規約（Rule）の更新**: `.agent/rules` 内の関連基準を修正。
- [ ] **Step 2: 仕組み（Agentic Tooling）の改修**: ビルドスクリプトやテンプレートを修正。
- [ ] **Step 3: 物理構造の適用**: ファイル移動やリネームを物理的に実行。

### Phase 3: Full Fidelity Validation (検証)
- [ ] `build_portal.ps1` を実行し、ポータルのビルドが成功することを確認。
- [ ] ブラウザ検証（Browser Pilot）等を用いて、意図通りの構造・表示になっているかを確認。

### Phase 4: Governance Hardening (固定と報告)
- [ ] 修正後の状態が「新標準」であることを宣言。
- [ ] 作業の証跡（成果物のパス、変更内容、検証結果）をワークフロー形式で報告。
- [ ] プロジェクトを次のフェーズ（あるいは継続作業）へ引き渡す。
